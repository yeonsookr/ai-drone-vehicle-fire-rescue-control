# Git 사용 및 협업 가이드라인 (`git_guide.md`)

본 문서는 `S15P11A302` 프로젝트의 일관된 코드 품질 유지와 원활한 협업을 위한 Git 브랜치 전략, 커밋 메시지 규칙, 그리고 풀 리퀘스트(PR) 절차를 정의합니다.

---

## 1. 브랜치 전략 (Branch Strategy)

우리는 **Git Flow**를 팀 환경에 맞게 단순화하여 사용합니다. 모든 작업은 개별 기능 단위 브랜치에서 진행한 뒤 검증을 거쳐 통합합니다.

### 1.1 브랜치 구조 및 명명 규칙

```text
main (배포)
  └── develop (개발 통합)
        ├── feature/server/COM-001 (기능 개발)
        ├── feature/ai/AI-001
        ├── feature/platform/PLT-004
        └── hotfix/server/BUG-102 (긴급 버그 수정)
```

| 브랜치 분류 | 대상 및 설명 | 브랜치 명명 규칙 |
| :--- | :--- | :--- |
| **`main`** | **최종 출시/배포**: 항상 빌드 가능하고 정상 동작하는 프로덕션 코드 상태를 유지합니다. | `main` |
| **`develop`** | **개발 통합**: 다음 릴리스를 위해 각 파트의 작업이 최종 통합되는 중심 브랜치입니다. | `develop` |
| **`feature`** | **기능 개발**: 새로운 기능 개발을 진행하는 브랜치로, 파트와 기능 ID를 명시합니다. | `feature/{part}/{기능ID}`<br/>*예: `feature/server/COM-001`* |
| **`hotfix`** | **긴급 수정**: 통합 또는 배포된 코드에서 발견된 중대 결함을 긴급히 수정할 때 사용합니다. | `hotfix/{part}/{결함ID_또는_설명}`<br/>*예: `hotfix/server/db-connection`* |

## 2. 커밋 메시지 규칙 (Commit Message Rules)

커밋 메시지는 Conventional Commits 명세와 기능 명세서의 **기능 ID**를 조합하여 변경 사항을 쉽게 추적할 수 있도록 합니다.

### 2.1 커밋 메시지 기본 포맷
```text
<type>(<scope>): <subject> [기능ID]

<body> (선택사항: 변경 원인 및 핵심 내용 서술)
```

### 2.2 커밋 타입 (`type`)
| 타입 | 설명 | 예시 |
| :--- | :--- | :--- |
| **`feat`** | 새로운 기능 추가 | `feat(server): MQTT 게이트웨이 인증 및 세션 검증 구현 [SEC-002]` |
| **`fix`** | 버그/오류 수정 | `fix(embedded): LiDAR 장애물 판정 거리 오차 공식 수정 [VEH-003]` |
| **`docs`** | 문서 변경 (코드 수정 없음) | `docs(docs): Git 사용 가이드라인 추가` |
| **`style`** | 코드 스타일 수정 (포맷팅, 세미콜론 누락 등) | `style(platform): 관제 대시보드 폰트 스타일 정돈` |
| **`refactor`** | 코드 리팩토링 (기능은 동일, 가독성/구조 개선) | `refactor(ai): YOLO 추론 루프 이미지 리사이즈 성능 개선` |
| **`test`** | 테스트 코드 추가 및 수정 | `test(server): 임무 생성 API 단위 테스트 케이스 추가 [MIS-001]` |
| **`chore`** | 빌드 설정 변경, 의존성 설치, 설정 파일 수정 | `chore(server): mysql 커넥터 라이브러리 버전 업데이트` |

### 2.3 작성 지침
- 제목은 **50자 이내**로 명확히 작성하고, 마침표(`.`)는 찍지 않습니다.
- 본문이 필요할 경우, 제목과 본문 사이에 **빈 행**을 한 줄 추가한 뒤 변경 배경과 영향 범위를 설명합니다.
- 반드시 대괄호안에 기능 명세서의 **기능 ID**(`[COM-001]`, `[AI-001]` 등)를 넣어 커밋과 요구사항의 관계를 추적할 수 있게 합니다.

## 3. 풀 리퀘스트 (PR) 및 코드 리뷰 규칙

- **직접 푸시 금지**: `main` 및 `develop` 브랜치에 직접 Push하는 행위는 보호 설정(Branch Protection)을 통해 금지합니다. 모든 변경은 PR을 통해서만 병합됩니다.
- **최소 승인(Approve) 요건**: 
  - 병합을 위해 최소 **1명 이상의 다른 파트 개발자** 혹은 **SI/QA(박연수)**의 리뷰 및 승인이 필요합니다.
  - 특히 연동이 엮인 부분(예: `Server` ↔ `Embedded` 연동)은 두 파트 담당자가 모두 승인한 후에만 병합합니다.
- **자동 빌드 및 검사**: PR 생성 시 빌드가 성공하고 모든 단위 테스트가 통과하는지 확인(CI 빌드 검증)해야 합니다.
