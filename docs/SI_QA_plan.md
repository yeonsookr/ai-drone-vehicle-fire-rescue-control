## 1. 시스템 기준

### 1.1 범위와 책임 경계

| 항목 | 기준 |
|---|---|
| 현장 통신 | `드론 내장 CPU ↔ 차량 Jetson Orin ↔ 관제 서버`의 2구간만 허용 |
| 사용자 명령 | `관제 사용자 → 관제 서버 → 차량 Jetson Orin → 드론` 순서로 전달하고 결과는 역순으로 회신 |
| MVP AI | 차량 Jetson Orin이 산불·연기·조난자 탐지와 사전 승인 범위의 로컬 명령 담당 |
| 관제 서버 | 정보 수집·저장·배포, 임무·권한·명령·ACK·감사 관리; MVP 서버 측 AI 추론 금지 |
| 실제 장비 | 드론 1대와 Jetson Orin 탑재 차량 1대의 1:1 페어 |
| 다중 관제 | Should; 복수 차량·드론 페어는 재현 가능한 mock data로 검증 |
| 독립 AI 서버 | Could; 관제 Backend하고만 통신하며 차량·드론 직접 연결·제어 금지 |

### 1.2 역할과 협업 창구

| 역할·담당자 | 책임 | SI·QA 확인 항목 |
|---|---|---|
| Embedded · 윤주한 | 드론 CPU·비행 제어기·센서, OrinCar·Jetson Orin·적재 거점, 현장 통신과 안전 정지 | 실장비 제약, fail-safe, 장비·구간별 시험 증거 |
| Backend · 변준성 | 관제 서버, API·DB·실시간 메시지, 임무·명령·ACK·보안·로그, 알고리즘과 mock 생성 | 서버 외부 계약, 상태·명령 이력, mock 재현성, 복구 절차 |
| Platform · 전우석 | 지도·영상·장비·임무·AI·경보·사용자 판단·명령·ACK UI | HTTP·WebSocket 계약, 사용자 판단 지점, 단절·복구 표시 |
| AI · 장민규 | Orin 모델 데이터·학습·평가·추론과 근거·성능; 확장 AI 서버 모델 | 검증 데이터셋, 정량 기준, 모델 버전·무결성, 낮은 신뢰도 처리 |
| SI 및 QA · 박연수 | 기준선·인터페이스·일정·리스크·형상, 통합·인수 시험, 운영 이관 | 추적표, 의사결정·변경 이력, 테스트·결함·인수 결과 |

### 2.2 추적 규칙

모든 작업은 다음 경로로 연결한다.

`기능 ID → 설계·인터페이스 → 개발 작업 → 테스트 케이스 → 결함 → 재시험 결과 → 인수 기록`

테스트 결과에는 최소한 실행일, 환경·버전, 입력 데이터, 로그·화면·영상, 기대 결과, 실제 결과, 판정자와 합격 여부를 남긴다.

## 5. 시험 실행과 합격 기준

### 5.1 실행 전 준비

- 기준 문서와 인터페이스 버전을 동결한다.
- 실제·mock·replay 여부와 장비·계정·모델·데이터 버전을 기록한다.
- 입력 데이터, 장애 주입 방법, 기대 결과와 증거 저장 위치를 준비한다.
- 안전 시험은 비상 정지와 장비 보호 수단을 먼저 확인한다.

### 5.2 최종 합격 기준

- Must 기능은 100% 통과하거나 미완료 항목마다 승인된 기준선 변경·인수 예외가 있어야 한다.
- 실제 1:1 페어의 두 MQTT 구간과 서버·웹 핵심 흐름을 실장비로 재시험한다.
- 핵심 기본 시나리오는 합의한 횟수만큼 연속 성공해야 한다. 권장 목표는 3회다.
- 미해결 Critical·Major 결함은 0건이어야 한다.
- 명령·AI 결과·사용자 판단·ACK가 공통 ID와 시각으로 끝까지 추적돼야 한다.
- 통신·AI·센서·배터리·장애물 실패 시 안전 동작과 복구 결과가 기록돼야 한다.
- mock은 계약·장애·다중 관제 증거로 사용할 수 있지만 실제 1:1 MVP 인수를 대체하지 않는다.

### 5.3 결함 등급

| 등급 | 기준 | 처리 원칙 |
|---|---|---|
| Critical | 인명·장비 위험, 비상 정지 실패, 시스템 전체 사용 불가 | 해결 전 시연·배포 중단 |
| Major | Must 실패, 데이터 유실, 핵심 E2E 진행 불가 | 완료 보류 후 현재 일정 내 수정·재시험 |
| Minor | 우회 가능한 오류, 일부 표시·성능 문제 | 영향과 일정에 따라 수정 |
| Trivial | 오탈자·경미한 사용성 개선 | 개선 목록에서 관리 |

결함 상태는 `NEW → TRIAGED → IN_PROGRESS → READY_FOR_RETEST → CLOSED`로 관리하며 재발하면 `REOPENED`로 되돌린다.
